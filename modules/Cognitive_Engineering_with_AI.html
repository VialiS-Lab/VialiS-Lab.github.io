단계활동산출물실행 주기
M: MonitorCSH/HAIS 지표 및 4개 Engine(KPI) 수집KPI Dashboard실시간/일간
I: Interpret이상 징후, 패턴, 병목 분석상태 진단 리포트주간
D: Decide개선 방안 선택 (모드 전환/루프 조정/정책 변경)실행 계획주간

🛠️ 도구: Cognitive System Health (CSH) & Human-AI Integration Score (HAIS)

최상위 KPI로, 전체 시스템의 건강 상태와 인간-AI 통합 수준을 정량화합니다.

CSH (Cognitive System Health)

```
CSH = (Working_Health × 0.20) + (Thinking_Health × 0.20) + (Knowledge_Health × 0.20) + (Learning_Health × 0.20) + (Governance_Health × 0.20)
```
구성 요소측정 지표정상 범위
Working_HealthQC 통과율, Workflow 성공률, 자동화율85% 이상
Thinking_HealthReasoning Confidence 평균, 가설 검증 성공률7.0/10 이상
Knowledge_HealthKnowledge Density, Reuse Rate, Growth Rate6.0/10 이상
Learning_HealthMastery Score 평균, Learning Health 지수70점 이상
Governance_Health정책 준수율, 예외 발생률, 신뢰도80% 이상

HAIS (Human-AI Integration Score)

```
HAIS = (Trust × 0.25) + (Automation × 0.20) + (DecisionQuality × 0.20) + (KnowledgeFlow × 0.20) + (Learning × 0.15)
```
구성 요소정의측정 방법
Trust인간의 AI에 대한 신뢰도자기 평가 설문
AutomationAI 자동화 수준자동화된 작업 비율
DecisionQualityAI 지원 의사결정 정확도실제 결과와 예측 일치율
KnowledgeFlow인간→AI, AI→인간 지식 전달 효율재사용률, 전이율
Learning시스템을 통한 학습 성과Mastery Score 평균

CSH/HAIS 해석 가이드

점수등급상태조치
8.0 ~ 10.0⭐⭐⭐⭐⭐최적유지 및 미세 조정
6.0 ~ 7.9⭐⭐⭐⭐양호일부 개선 필요
4.0 ~ 5.9⭐⭐⭐보통중간 개선 필요
2.0 ~ 3.9⭐⭐미흡대대적 개선 필요
0.0 ~ 1.9⭐위험시스템 재설계 필요

인지 상태설명AI 동작 모드활성화 조건
Creative Mode새로운 아이디어, 브레인스토밍Generator 강화: 다양한 대안, 자유로운 연결, 확산적 사고CSH Creative 지표 상승 / 사용자 요청

🛠️ 도구: Closed Human-AI Interaction Loop

v1의 단순 피드백을 넘어, Human → AI → Correction → Update → Validation → Knowledge의 완전한 순환 루프를 설계합니다.

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                    CLOSED HUMAN-AI INTERACTION LOOP                             │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  [1] Human Input (인간의 요청/질문/아이디어)                                   │
│        ↓                                                                       │
│  [2] AI Generate (AI가 초안/대안/분석 생성) — Generator 모드                  │
│        ↓                                                                       │
│  [3] Human Correction (인간이 수정/보완/피드백 제공)                           │
│        ↓                                                                       │
│  [4] AI Update (AI가 피드백 반영하여 수정) — Critique/Evidence 모드           │
│        ↓                                                                       │
│  [5] Human Validation (인간이 최종 결과 검증 및 승인) — Decision 모드         │
│        ↓                                                                       │
│  [6] Knowledge Update (최종 결과를 K-OS에 저장, Learning에 반영)              │
│        ↓                                                                       │
│  🔄 → [1]로 순환 (새로운 요청/개선된 맥락으로)                                  │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

📌 Module 1: 맥락을 설계하자 (Context)

C.E.N.G.I.N.E.의 첫 번째 단계. 인간과 AI가 함께 일하는 환경의 경계, 조건, 목적을 정의합니다.

① Experiment — 맥락의 중요성

당신이 해야 할 일: 동일한 질문을 두 가지 맥락에서 AI에게 던져보세요.

맥락 A (맥락 없음):

"이 보고서를 검토해줘."

맥락 B (맥락 있음):

"이 보고서는 신규 고객사에 제출할 제안서입니다. 의사결정자는 CEO이며, 핵심 관심사는 ROI와 실행 가능성입니다. 기술적 깊이보다는 비즈니스 가치를 강조해야 하며, 검토 시 ① 빠진 내용, ② 과장된 주장, ③ 설득력을 높일 부분을 집중적으로 봐줘."

→ AI의 답변 품질은 어떻게 달라졌나요?

② Reflection — 생각해보기

  • 맥락이 AI의 답변 품질에 얼마나 큰 영향을 미쳤나요?
  • AI에게 맥락을 제공하지 않을 때 어떤 문제가 발생하나요?

③ Discovery — 방금 깨달은 것

"아, 좋은 AI 협업은 좋은 맥락 설계에서 시작되는구나. AI는 내가 생각하는 것보다 훨씬 더 많은 맥락을 필요로 해."

④ Rule — C.E.N.G.I.N.E. 법칙

법칙내용
Rule C1맥락이 없으면 AI는 추측한다. 추측은 오류로 이어집니다.
Rule C2맥락은 5가지 요소를 포함한다. ① 목적(왜?), ② 대상(누구를 위해?), ③ 상황(어떤 조건?), ④ 기준(무엇을 중요하게?), ⑤ 경계(무엇을 하지 말아야?).
Rule C3맥락은 공유되어야 한다. 인간과 AI가 동일한 맥락을 공유해야 일관된 협업이 가능합니다.

⑤ Practice — Context Design Document 작성

Step 1. AI_Work/cognitive/context/ 폴더를 만드세요.

Step 2. 자신의 업무에서 AI와 협업하는 대표적인 상황 1개를 선택하세요.

Step 3. Context Design Document를 작성하세요.

```markdown
# Context Design Document
```

① Experiment — Closed Loop vs Open Loop

당신이 해야 할 일: 두 가지 상호작용 방식을 비교해보세요.

Open Loop (열린 루프):

"초안을 작성해줘." → AI가 초안 작성 → 종료

Closed Loop (닫힌 루프):

"초안을 작성해줘. 초안이 나오면 내가 피드백을 줄게. 피드백을 반영해서 수정하고, 내가 다시 검토할게. 최종 승인 후에는 Knowledge에 저장해줘."

→ 어떤 방식이 최종 결과물의 품질을 더 높였나요?

    • Monitor: CSH 하락 감지
    • Interpret: 원인 분석
    • Decide: 개선 방안 선택
    • Act: 개선 실행
    • Review: CSH 재측정

→ CSH가 회복되었나요?

② Reflection — 생각해보기

  • 시스템이 스스로 문제를 진단하고 개선한다면, 어떤 점이 편리할까요?
  • Self-Optimization이 없을 때는 어떻게 문제를 해결했나요?

③ Discovery — 방금 깨달은 것

"아, 좋은 시스템은 스스로를 진단하고 치유하는구나. MIDAR가 없으면 병들어도 모르고 있었을 거야."

④ Rule — C.E.N.G.I.N.E. 법칙

법칙내용
Rule N1시스템은 Self-Optimization을 통해 진화한다. MIDAR Loop가 시스템의 지속적 개선을 보장합니다.
Rule N2Self-Optimization은 KPI 기반으로 작동한다. CSH/HAIS가 임계치를 하회하면 자동으로 개선 방안이 실행됩니다.
Rule N3진화는 단계적으로 이루어진다. ① 안정화(현재 유지), ② 최적화(성능 개선), ③ 혁신(구조 변경).
